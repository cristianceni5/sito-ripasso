1. Introduzione al C++ e Differenze con il C

Il C++ è un'estensione del linguaggio C creata da Bjarne Stroustrup. Introduce il paradigma della Programmazione Orientata agli Oggetti (OOP) mantenendo la retrocompatibilità (quasi totale) col C.

Compilazione Multi-file (E0b)

Un progetto C++ reale separa il codice in più file (.h dichiarazione, .cpp implementazione). Il linker unisce gli object file in un eseguibile. Bisogna sempre includere le parentesi graffe {} per un metodo, anche se vuoto, altrimenti il linker segnala errori.

// E0b_compilation_example/main.cpp
#include <iostream>
#include <cstdlib>
#include "greeter.h"
#include "adder.h"

int main() {
    sayHello("Joe");
    int *array;
    int arraySize = 10;
    array = static_cast<int*>(std::malloc(arraySize * sizeof(int)));
    for (int i = 0; i < arraySize; i++)
        array[i] = i;
    std::cout << sum(array, arraySize) << std::endl;
    std::free(array);
    return 0;
}

Namespace e Scope Resolution

Il C++ introduce i namespace per evitare conflitti di nomi. L'operatore :: è detto scope resolution operator. Un nome preceduto dal suo namespace si chiama qualified name (es. std::cout). Il Name Mangling permette al compilatore di risolvere nomi unici per entità con lo stesso identificatore ma firme diverse (overloading).

2. Classi e Oggetti — ADT e Incapsulamento

Una Classe è un "progetto" (un tipo definito dall'utente). Un Oggetto è un'istanza concreta in memoria. Ogni classe definisce attributi (variabili membro) e metodi (funzioni membro).

ADT (Abstract Data Type): Un tipo di dato definito dal suo comportamento (interfaccia pubblica), nascondendo l'implementazione interna (Information Hiding). Garantisce maggiore certezza e modularità del codice.

Esempio completo: Weapon (E1)

Dalla prima esercitazione: una classe con costruttore, getter, setter e attributi protetti.

// Weapon.h (E1_class_exercise)
class Weapon {
public:
    Weapon(int s = 0, bool magic = false) : strength(s), magic(magic) {
        if (strength <= 0)
            strength = 1;
    }

    int use();                     // Dichiarato qui, definito in Weapon.cpp

    // Getter
    int getStrength() { return strength; }
    bool isMagic() { return magic; }

    // Setter (con validazione!)
    void setStrength(int s) { if (s > 0) strength = s; }
    void setMagic(bool m) { magic = m; }

protected:
    int strength;
    bool magic;
};
// Weapon.cpp (implementazione separata)
#include "Weapon.h"

int Weapon::use() {
    if (magic)
        strength = strength * 2; // XXX rischio int overflow!
    return strength;
}

GameCharacter completo (E1)

Il personaggio del gioco RPG: costruttore con valori di default, overloading di move(), uso di this, e combattimento con Weapon.

// GameCharacter.h (E1_class_exercise)
class GameCharacter {
public:
    GameCharacter(int hp = 10, int a = 10);

    // Overloading: stesso nome, parametri diversi
    void move(int x, int y);
    void move(int distance);

    bool fight(GameCharacter &enemy);
    int receiveDamage(int points);

    // Getter & Setter
    int getHP();
    void setHP(int hp);
    Weapon* getWeapon();
    void setWeapon(Weapon* weapon);

private:
    int HP;
    int armor;
    int posX, posY;
    Weapon* weapon;
};
// GameCharacter.cpp - Costruttore con lista di inizializzazione
GameCharacter::GameCharacter(int hp, int a) : HP(hp), armor(a), posX(0), posY(0), weapon(nullptr) {
    if (hp <= 0) HP = 10;
    if (a <= 0) armor = 10;
}

// fight() usa il puntatore weapon e this->
bool GameCharacter::fight(GameCharacter &enemy) {
    int hit = 1;
    if (weapon) {
        hit = weapon->use();
    }
    int damage = 0;
    if (hit > enemy.armor) {
        damage = receiveDamage(hit);
    }
    return (damage > 0);
}

3. Costruttori in Dettaglio

Tipi di Costruttore

  • Costruttore di default: Senza argomenti. Generato automaticamente dal compilatore solo se non ne dichiariamo nessun altro.
  • Costruttore parametrico: Con argomenti. Può avere valori di default.
  • Costruttore di copia (Copy Constructor): Prende un riferimento const allo stesso tipo. Generato automaticamente dal compilatore (shallow copy).
  • explicit Constructor: Un costruttore con un solo argomento va sempre marcato explicit per prevenire conversioni implicite indesiderate. È una norma anche tra i programmatori C++ di Google.
  • Delegating Constructor: Un costruttore che invoca un altro costruttore della stessa classe nella lista di inizializzazione.

Lista di Inizializzazione

La sintassi : membro1(val1), membro2(val2) è la lista di inizializzazione ed è preferibile all'assegnamento nel corpo — è più efficiente e l'unico modo per inizializzare const e riferimenti.

// ✅ Costruttore con lista di inizializzazione
GameCharacter::GameCharacter(int hp, int a) : HP(hp), armor(a), posX(0), posY(0) {}

// ✅ explicit su costruttore a singolo argomento
explicit Sword(int s, bool m = false, bool v = false);

// ✅ Delegating Constructor
Weapon() : Weapon(1, false) {}  // Delega al costruttore parametrico

// C++11: inizializzazione in-class
class X {
public:
    int i = 4;
    int j{5}; // Sintassi alternativa con brace-init
};

4. Distruttore, this, static e friend

Distruttore (~NomeClasse())

  • Chiamato automaticamente alla distruzione dell'oggetto (fine scope o delete).
  • Il suo scopo è pulizia (liberare risorse: file, memoria heap, connessioni), non deallocare la memoria dell'oggetto stesso.
  • Lo stack viene pulito automaticamente. Il delete serve solo per memoria allocata con new.
  • Viene generato automaticamente dal compilatore con corpo vuoto.
// E3 - Distruttore esplicito per risorse dinamiche
GameCharacter::~GameCharacter() {
    delete weapon;  // Libera la memoria dell'arma allocata con new
}

Il puntatore this

Ogni oggetto contiene un puntatore implicito a sé stesso: this. È di tipo NomeClasse*. Serve per:

  • Distinguere attributi da parametri con lo stesso nome: this->HP = hp;
  • Passare l'indirizzo dell'oggetto corrente ad altri oggetti (inversione di responsabilità)
  • Ritornare l'oggetto corrente: return *this;
void GameCharacter::setHP(int hp) {
    this->HP = hp;  // this-> disambigua dal parametro
}

static Members

  • Un membro static ha una sola copia condivisa tra tutti gli oggetti della classe.
  • Un metodo statico non può accedere a membri non-statici (potrebbero non essere inizializzati). Si invoca con Classe::metodo().
  • In C++17 si può fare static inline direttamente nel file .h.
// E3 - metodo statico per calcolare distanza (non ha bisogno di un oggetto)
static int l1Distance(const GameCharacter& p, const GameCharacter& q);

// Chiamata:
GameCharacter::l1Distance(hero, enemy);

friend

Una funzione o classe friend può accedere ai membri private di un'altra classe. Rompe l'incapsulamento — da usare con molta cautela (tipicamente per I/O overloading).

class Point {
private:
    int x, y;
public:
    friend bool operator==(Point a, Point b);
    Point() : x(0), y(0) {}
};

bool operator==(Point a, Point b) {
    return (a.x == b.x && a.y == b.y); // Accede a x,y privati
}

5. const Correctness

La keyword const in C++ è molto più potente di #define e va usata sistematicamente.

Regole del const con i puntatori

const char* p = greeting;        // Puntatore a dato costante (posso solo leggere)
char* const p = greeting;        // Puntatore costante (punta sempre allo stesso indirizzo)
const char* const p = greeting; // Tutto costante

Metodi const

  • Se un metodo è dichiarato const, promette di non modificare l'oggetto. Così può essere chiamato su oggetti costanti.
  • Si può fare overloading solo con const: se esistono due versioni dello stesso metodo (una const e una no), il compilatore sceglie in base al tipo dell'oggetto.
  • La keyword mutable rilassa il const: un membro mutable può essere modificato anche da metodi const (es. cache, lazy evaluation).
// Getter const (dall'E3): promette di non modificare l'oggetto
int getHP() const;
int getArmor() const;
bool isLegalFight(const GameCharacter &enemy) const;

// Overloading con const
const char& operator[](size_t pos) const { ... }  // Versione read-only
      char& operator[](size_t pos) { ... }        // Versione read-write

6. Shallow Copy vs Deep Copy & Regola dei Tre

Metodi generati dal compilatore

Il compilatore genera automaticamente: costruttore di copia, operatore di assegnamento (operator=) e distruttore. Operano una shallow copy (copia bit-a-bit).

// Metodi impliciti generati dal compilatore:

// 1. Copy constructor
GameCharacter(GameCharacter& that) : name(that.name), hp(that.hp) {}

// 2. Copy assignment operator
GameCharacter& operator=(GameCharacter& that) {
    name = that.name;
    hp = that.hp;
    return *this;
}

// 3. Destructor
~GameCharacter() {}

Problema della Shallow Copy con i Puntatori

Con la copia superficiale, i puntatori vengono copiati bit-a-bit: la copia e l'originale puntano alla stessa memoria. Se l'originale viene distrutto, il distruttore dealloca la risorsa, e la copia punta a memoria non più valida → crash.

class GameCharacter {
public:
    ~GameCharacter() {
        delete[] inventory; // Dealloca l'inventario
    }
private:
    std::string name;
    int hp;
    Potion* inventory;  // ⚠️ Shallow copy copia il puntatore, non i dati!
};
// Se copio hero in hero2, entrambi puntano allo STESSO inventory.
// Quando hero viene distrutto, hero2.inventory punta a memoria deallocata!

Regola dei Tre (Rule of Three)

Se una classe gestisce risorse (memoria, file, socket...) allora bisogna implementare esplicitamente tutti e tre:

  1. Copy Constructor (deep copy)
  2. Copy Assignment Operator (deep copy)
  3. Destructor (rilascio risorse)
// E3: espliciti nel .h per gestire il puntatore weapon
GameCharacter(const GameCharacter &original);       // Deep copy constructor
GameCharacter &operator=(const GameCharacter &right); // Deep copy assignment
~GameCharacter();                                    // Destructor

Disabilitare la copia (= delete)

A volte un oggetto non deve essere copiabile (es. singleton, risorse di sistema). Si usa = delete:

Foo& operator=(const Foo&) = delete;  // Disabilita l'assegnamento

7. Overloading degli Operatori

Permette di ridefinire il funzionamento degli operatori standard (+, =, <<, ==, []) per i nostri tipi di dato. In questo modo si possono usare gli oggetti in modo naturale e intuitivo.

// Overloading di operator== con friend
class Point {
private:
    int x, y;
public:
    friend bool operator==(Point a, Point b);
};

bool operator==(Point a, Point b) {
    return (a.x == b.x && a.y == b.y);
}

// Overloading di operator= (copy assignment) dalla Rule of Three
GameCharacter& operator=(const GameCharacter& that) {
    name = that.name;
    hp = that.hp;
    return *this;
}

8. Ereditarietà (Inheritance)

Permette a una nuova classe (Derived) di ereditare proprietà e metodi da una classe base. È una relazione "Is-A" ("È un"). In C++ esiste anche l'ereditarietà multipla.

Tipi di Ereditarietà

Ereditarietà pubblica è la più comune: ci interessa l'interfaccia della classe base.

Ereditarietà vs Composizione

Liskov Substitution Principle

Un oggetto derivato deve poter sostituire il suo tipo base senza rompere il programma. Funziona solo in una direzione: se Student : public Person, allora eat(Student) funziona, ma study(Person) no. Esempio critico: Se Penguin : Bird e Bird ha fly(), il pinguino viola il principio perché non può volare.

Costruttori ed Ereditarietà

Il costruttore della classe derivata invoca il costruttore della base nella lista di inizializzazione. La catena di distruzione procede dalla derivata verso la base.

// Sword estende Weapon: chiama il costruttore della base
Sword::Sword(int s, bool m, bool v) : Weapon(s, m), valirian(v) {
    // Il corpo può restare vuoto
}

// Bow: costruttore inline con delegation alla base
explicit Bow(int s, bool m = false, int a = 20) : Weapon(s, m), arrows(a) {}

9. Polimorfismo e Virtual/Override (Binding Dinamico)

Il Problema del Binding Statico

Regola del Binding (comportamento default del compilatore):

  1. Se ho una variabile su cui chiamo un metodo, il compilatore chiama il metodo del tipo della variabile (non dell'oggetto reale).
  2. Se ho un puntatore, sarà chiamato il metodo del tipo del puntatore, non dell'oggetto puntato.

Nel metodo fight() di GameCharacter si chiama weapon->use(). Ma weapon è un puntatore a Weapon*: senza virtual, chiamerebbe sempre Weapon::use(), anche se l'oggetto reale è un Sword o un Bow!

virtual e override

Marcando il metodo come virtual nella classe base, si ritarda la decisione all'esecuzione (late binding). Quando un metodo è virtual, rimane tale per tutta la gerarchia. Le classi derivate usano override per dichiarare esplicitamente la riscrittura.

// Weapon.h (E3) - metodo virtual nella base
class Weapon {
public:
    virtual int use();  // Virtual: il binding è rimandato a runtime
protected:
    int strength;
    bool magic;
};

// Sword.h - override esplicito
class Sword : public Weapon {
public:
    virtual int use() override;  // Riscrive use() della base
protected:
    bool valirian;
};

Implementazione dell'Override (Sword)

// Sword.cpp - override di use()
// PRIMA chiama il metodo della base, POI aggiunge il comportamento specifico
int Sword::use() {
    int result = Weapon::use();  // Chiama la versione base
    if (valirian)
        result *= 2;              // Raddoppia il danno se valeriana
    return result;
}

La gerarchia RPG completa (E3)

// Gerarchia delle Armi:
Weapon (base - virtual use())
  ├── Sword : public Weapon   // override use(), aggiunge valirian
  └── Bow   : public Weapon   // override use(), aggiunge arrows

// Gerarchia dei Personaggi:
GameCharacter (base)
  ├── Knight   // dexterity, paladin, override fight()/move()
  ├── Wizard   // mana, name, override fight(), virtual doMagic()
  ├── Orc      // specializzazione nemica
  └── Skeleton // specializzazione nemica

Regole chiave per override

  • override va messo solo nella dichiarazione (.h), mai nella definizione (.cpp).
  • Il compilatore verifica che esista effettivamente un metodo virtuale con la stessa firma nella classe base.
  • La scelta di quali metodi devono essere virtuali è una decisione architetturale importantissima.

10. Principi di Design OOP

  • Single Responsibility Principle (SRP): Ogni classe ha una sola responsabilità. Se ha più "cose da fare", va scomposta. Trucco: se il nome della classe non descrive bene cosa fa un metodo, quel metodo probabilmente non appartiene a quella classe.
  • Open/Closed Principle (OCP): Aperto all'estensione, chiuso alla modifica. L'ereditarietà e il polimorfismo supportano questo principio.
  • Liskov Substitution Principle (LSP): Un oggetto derivato deve poter sostituire la sua base senza errori (vedi sopra: il caso Penguin/Bird).
  • DRY (Don't Repeat Yourself): Evitare duplicazione di codice.
  • Coesione alta, Accoppiamento basso: Le classi devono fare poche cose ma bene (coesione) ed essere il più indipendenti possibile (accoppiamento basso).

11. enum class (C++11)

Il C++11 introduce enum class (detto scoped enum) come alternativa tipizzata e sicura alla vecchia enum del C. I valori non inquinano il namespace e non si convertono implicitamente a int.

// E2 / E3 - GameEvent come enum class
enum class GameEvent {
    quit, left, up, down, right, fight, noop
};

// I valori DEVONO essere qualificati con il nome dell'enum
GameEvent e = GameEvent::fight;   // ✅ corretto
// GameEvent e = fight;            // ❌ errore di compilazione

// TileType per il Dungeon (E3)
enum class TileType {
    Unused = 0, DirtWall, DirtFloor, StoneWall,

12. Classe Dice: static, srand e const come membro

La classe Dice (E2/E3) è un ottimo esempio pratico di utilizzo avanzato di: membro static condiviso tra tutte le istanze, membro const inizializzato nella lista, e uso di srand/rand con re-seeding controllato.

// Dice.h (E2_class_exercise)
class Dice {
public:
    explicit Dice(int f = 6) : faces(f) {
        if (faces < 2) faces = 6;
        srand(time(0));   // Inizializza il generatore casuale
    }

    int roll(int r);

private:
    int faces;
    static int numRoll;         // UNA sola copia per tutte le istanze
    const int NUM_MAX = 6969;  // Costante per-oggetto inizializzata in-class
};
// Dice.cpp - il membro static va DEFINITO fuori dalla classe
int Dice::numRoll = 0;

int Dice::roll(int r) {
    int result = 0;
    // Se il dado è stato lanciato troppo spesso, re-seed il RNG
    if (numRoll > NUM_MAX) {
        srand(time(0));
        numRoll = 0;
    }
    for (int i = 0; i < r; i++)
        result += rand() % faces;
    return result;
}

Nota importante su srand: Chiamare srand(time(0)) ripetutamente in poco tempo assegna lo stesso seed (perché time() ha risoluzione al secondo) producendo la stessa sequenza. Per questo si usa un contatore statico numRoll che triggera il re-seed solo ogni NUM_MAX lanci.

Dice in GameCharacter.cpp (E3)

Il metodo fight() usa un dado a 6 facce per introdurre casualità nel combattimento:

int GameCharacter::fight(GameCharacter &enemy) {
    Dice myDice(6);                // Dado a 6 facce, locale alla funzione
    int hit = (weapon) ? weapon->use() : 1;

    int hitChance = myDice.roll(2);  // Tira 2 dadi
    if (hitChance > 4) {             // Colpo riuscito se somma > 4
        hit += myDice.roll(1);       // Danno bonus casuale
        if (hit > enemy.getArmor()) {
            return enemy.receiveDamage(hit);
        }
    }
    return 0;  // Nessun danno
}

13. Implementazione Completa della Deep Copy (E3)

In E3, GameCharacter gestisce un puntatore a Weapon*. La shallow copy del compilatore causerebbe doppia deallocazione. Ecco l'implementazione completa della Rule of Three:

// Copy Constructor: deep copy del puntatore weapon
GameCharacter::GameCharacter(const GameCharacter &original) {
    HP = original.HP;
    posX = original.posX;
    posY = original.posY;

    if (original.weapon != nullptr)
        weapon = new Weapon(*original.weapon);  // Alloca una NUOVA arma
    else
        weapon = nullptr;
}

// Copy Assignment Operator: gestisce self-assignment e dealloca il vecchio
GameCharacter &GameCharacter::operator=(const GameCharacter &right) {
    if (this != &right) {             // ⚠️ Self-assignment check!
        if (weapon != nullptr)
            delete weapon;          // Libera la vecchia arma

        HP = right.HP;
        posX = right.posX;
        posY = right.posY;

        if (right.weapon != nullptr)
            weapon = new Weapon(*right.weapon); // Copia profonda
        else
            weapon = nullptr;
    }
    return *this;  // Permette a = b = c
}

// Destructor: libera l'arma solo se esiste
GameCharacter::~GameCharacter() {
    if (weapon != nullptr)
        delete weapon;                // Evita delete su nullptr
}

Il Self-Assignment Check (if (this != &right)) è critico! Senza di esso, se scrivi hero = hero;, il codice prima dealloca weapon e poi cerca di copiarlo: crash immediato.

14. Specializzazioni dei Personaggi (E3)

L'esercitazione E3 costruisce una gerarchia di classi derivate da GameCharacter, ognuna con comportamenti specifici. Questo è polimorfismo in azione.

Knight (Cavaliere)

Ha destrezza (dexterity) e può essere un paladino (paladin). Override di fight() (secondo attacco se destrezza > 10, +10 danno se paladino) e di move() (+1 movimento se destrezza > 10).

// Knight.h
class Knight : public GameCharacter {
public:
    char getCharacterSymbol() const { return 'K'; }
    // override fight(): se dexterity > 10, secondo attacco; +10 danno se paladin
    // override move():  se dexterity > 10, +1 movement
private:
    int dexterity;
    bool paladin;
    std::string name;
};

Wizard (Mago)

Ha mana e un nome. Override di fight() che usa il mana per cambiare il comportamento. Introduce doMagic() come nuovo metodo virtual (può essere ulteriormente esteso da sottoclassi di Wizard).

// Wizard.h
class Wizard : public GameCharacter {
public:
    char getCharacterSymbol() const { return 'W'; }
    virtual void doMagic();  // Nuovo metodo, virtual per ulteriori estensioni
    // override fight(): usa mana per modificare behaviour
protected:
    int mana;
    std::string name;
};

Orc (Nemico)

Nemico con forza bruta. Override di fight() (secondo attacco se forza > 10) e move().

// Orc.h
class Orc : public GameCharacter {
public:
    explicit Orc(int s, int hp=20);
    char getCharacterSymbol() { return 'O'; }
    // override fight(): se strength > 10, secondo attacco
protected:
    int strength;
};

Skeleton (Nemico)

Nemico resistente alla magia. Override di receiveDamage(): se magicRecover è true, il danno magico viene ridotto.

// Skeleton.h
class Skeleton : public GameCharacter {
public:
    char getCharacterSymbol() const { return 'S'; }
    // override receiveDamage(): magicRecover riduce il danno magico
private:
    bool magicRecover;
};

16. Game Loop Pattern e Architettura dell'Applicazione (E2/E3)

Il progetto RPG demonstra un pattern fondamentale dell'architettura software: il Game Loop. Il loop ripete all'infinito: poll input → update stato → render. Vedi gameprogrammingpatterns.com/game-loop.html.

// main.cpp (E2/E3) - Struttura del Game Loop
int main() {
    // 1. SETUP: crea mappa, personaggi, armi
    Dungeon map;
    map.createDungeon(80, 25, 50);

    GameCharacter* hero;  // Puntatore base → polimorfismo
    // hero = new Knight(...) o new Wizard(...)

    Weapon* sword;
    // sword = new Sword(10) o new Bow(8)
    hero->setWeapon(sword);

    // 2. RENDER iniziale
    renderHUD(*hero);
    renderGame(map, *hero, *enemy);

    // 3. GAME LOOP infinito
    while (true) {
        GameEvent event = getEvent();         // Poll: legge tastiera
        bool quit = updateGame(event, *hero, *enemy, map); // Update
        if (quit) return 0;
        renderHUD(*hero);
        renderGame(map, *hero, *enemy);        // Render
    }
}

Separazione delle Responsabilità (SRP in pratica)

Il codice di E2/E3 mostra SRP "in the wild": ogni funzione ha un unico scopo.

FunzioneResponsabilità
getEvent()Solo lettura da tastiera, mappa char a GameEvent
isLegalCell()Solo: questa cella è percorribile?
isLegalMove()Solo: questo movimento è legale? (usa isLegalCell + check nemici)
updateGame()Solo: applica l'evento al game state
renderGame()Solo: disegna a schermo il game state corrente
renderHUD()Solo: stampa HP, armor, weapon strength dell'eroe

Dichiarazioni e Inizializzazioni

Dichiarazione variabileTipo nome;Riserva spazio in memoria per la variabile. Se è locale (sullo stack) NON viene inizializzata automaticamente: il suo valore è spazzatura (garbage). Inizializzazione con =int x = 5;Sintassi "tradizionale" C. Chiama il costruttore di copia (prima C++11). Con i tipi primitivi è equivalente all'inizializzazione diretta.

Operatori

CategoriaOperatoriSemantica e Precedenza
Aritmetici+ - * / %/ su interi esegue divisione intera con troncamento verso zero: 7/2 == 3. % è il resto: 7%2 == 1. Per la divisione floating point almeno uno degli operandi deve essere float/double: 7.0/2 == 3.5.
Confronto== != < > <= >=Restituiscono bool. ATTENZIONE: == non è uguale a =. Scrivere if (x = 5) al posto di if (x == 5) è un errore logico silenzioso (assegna 5 a x e valuta come true). La convenzione "Yoda" if (5 == x) previene questo.
Logici&& || !Short-circuit evaluation: in A && B, se A è false B non viene valutato. In A || B, se A è true B non viene valutato. Questo permette pattern come if (p != nullptr && p->value > 0) senza rischio di null dereference.
Bit a bit& | ^ ~ << >>Operano sui singoli bit della rappresentazione binaria. << n equivale a moltiplicare per 2n (velocissimo). >> n divide per 2n. Usati per flag di stato, maschere, ottimizzazioni.
Assegnamento composto+= -= *= /= %=x += 5 è equivalente a x = x + 5 ma viene valutato una sola volta (importante se x ha side effects). Più leggibile e preferibile.
Incremento/Decremento++ -- (pre e post)Pre-incremento (++i): incrementa e restituisce il nuovo valore. Post-incremento (i++): restituisce il vecchio valore, poi incrementa. Per oggetti complessi (iteratori), ++i è più efficiente perché non crea una copia temporanea.
Ternariocond ? a : bSe cond è vera, l'espressione vale a, altrimenti b. Entrambi i rami devono avere tipi compatibili. NON usare per espressioni con side effects complessi: riduce la leggibilità.
Scope (::)Classe::membroScope Resolution Operator. Accede a un membro di un namespace, classe o enum class. Usato anche per definire metodi fuori dalla classe nel .cpp: void Weapon::use(). Se preceduto da nulla (::nome), indica il namespace globale.
Freccia (->)ptr->membroEquivalente a (*ptr).membro. Accede a un membro attraverso un puntatore. Se ptr è nullptr, causerà un crash (undefined behavior). Usare sempre il null check prima: if (weapon) weapon->use();.
Sizeofsizeof(tipo) o sizeof exprOperatore evaluato a compile time (non a runtime) che restituisce la dimensione in byte del tipo o dell'espressione. Non valuta l'espressione: sizeof(x++) non incrementa x. Ritorna un valore di tipo size_t (intero senza segno).
Cast espliciti (C++)static_cast<T>(x)static_cast: conversione controllata a compile-time. È il cast sicuro preferito in C++. dynamic_cast: cast a runtime su gerarchia di classi con funzioni virtuali (verifica se la conversione è valida). reinterpret_cast: cast pericoloso, interpreta la sequenza di bit in modo diverso. const_cast: rimuove/aggiunge const. I cast in stile C ((int)x) sono sconsigliati perché non esplicitano le intenzioni.

Strutture di Controllo

CostruttoSintassiSemantica dettagliata
if / else if / elseif (cond) { ... } else { ... }Le parentesi graffe sono facoltative se il corpo è una sola istruzione, ma omettere le graffe è una pratica rischiosa: il famoso bug "goto fail" di Apple (2014) fu causato da un if senza graffe con indentazione ingannevole. Il corso raccomanda sempre di usare le graffe.
switchswitch(valore) { case X: ...; break; default: ... }Confronta un valore intero/enum con costanti. Senza break c'è il fall-through: l'esecuzione prosegue nel case successivo (a volte intenzionale, spesso bug). Con enum class il compilatore può avvisare di case non gestiti se si omette il default.
whilewhile (cond) { ... }Valuta la condizione prima del corpo. Se la condizione è falsa sin dall'inizio, il corpo non viene mai eseguito. Attenzione ai loop infiniti: assicurarsi che la condizione diventi falsa ad un certo punto (o usare break).
do-whiledo { ... } while (cond);Valuta la condizione dopo il corpo. Il corpo viene eseguito almeno una volta. Utile per menu interattivi (almeno una volta si mostra il menu). Nota il ; obbligatorio dopo la parentesi.
for classicofor (init; cond; step) { ... }Le tre parti sono tutte opzionali: for(;;) è un loop infinito. La variabile di controllo dichiarata nell'init (int i = 0) è visible solo all'interno del for (scoping). L'init viene eseguita una volta; cond e step ad ogni iterazione.
Range-based for (C++11)for (auto& elem : collezione) { ... }Itera su tutti gli elementi di un container (array, std::vector, std::string...). auto& evita la copia (usa riferimento). const auto& se non si deve modificare. Non usare quando si deve anche accedere all'indice: in quel caso preferire il for classico.
break / continuebreak; / continue;break: esce immediatamente dal loop o switch più interno. continue: salta il resto del corpo e passa alla prossima iterazione (valutando di nuovo la condizione). In un for, continue esegue comunque il step.
returnreturn valore;Termina la funzione corrente e ritorna il valore al chiamante. Il tipo del valore deve essere compatibile con il tipo di ritorno dichiarato. return; in una funzione void è opzionale ma valido. Eseguire una funzione non-void senza return è undefined behavior.

Puntatori e Riferimenti

CostruttoSintassiSemantica dettagliata
Puntatoreint* p = &x;Una variabile che contiene un indirizzo di memoria. &x è l'operatore "address-of": prende l'indirizzo della variabile x. *p è il "dereference": accede al valore puntato. I puntatori possono essere nullptr, possono cambiare ciò che puntano, e possono essere aritmeticamente incrementati (pointer arithmetic). Questa flessibilità li rende potenti ma pericolosi.
Riferimento (Reference)int& r = x;Un riferimento è un alias della variabile originale: non crea una copia, punta allo stesso oggetto in memoria. Differenze chiave con i puntatori: (1) deve essere inizializzato alla dichiarazione; (2) non può essere riassegnato per puntare ad altro; (3) non può essere null; (4) non ha sintassi di dereferenziamento (si usa r direttamente come se fosse x). Nei parametri di funzione, passare per riferimento evita la copia.
Riferimento constconst int& r = x;Riferimento a sola lettura. Può essere legato anche a temporanei e letterali, a differenza di int&. Idioma classico per parametri funzione: void foo(const std::string& s) evita la copia della stringa ma garantisce che la funzione non la modifichi. Il lifetime del temporaneo è esteso per tutta la vita del riferimento const.
Puntatore a constconst int* p = &x;Il dato puntato è costante; il puntatore può essere riassegnato. *p = 5 è illegale. p = &y è legale. Mnemonica: leggere da destra a sinistra rispetto a *: "puntatore a int-costante".
Puntatore constint* const p = &x;Il puntatore è costante; il dato puntato è modificabile. p = &y è illegale. *p = 5 è legale. Usato tipicamente per array passati a funzioni (l'indirizzo base non cambia).
Allocazione dinamicaint* p = new int(5);new alloca memoria sull'heap (non sullo stack) e chiama il costruttore. La memoria rimane valida finché non si chiama delete esplicitamente, indipendentemente dallo scope. Memory leak: se si perde il puntatore prima di chiamare delete, la memoria non viene più liberata per tutta la durata del programma.
Deallocazionedelete p; delete[] arr;delete chiama il distruttore e libera la memoria heap. delete[] va usato per array allocati con new[]. Chiamare delete su un puntatore già deallocato (double free) è undefined behavior. Dopo il delete, mettere il puntatore a nullptr per sicurezza.
Array staticiint arr[10];Array a dimensione fissa allocato sullo stack. La dimensione deve essere una costante a compile-time. In C++ moderno preferire std::array<int, 10> o std::vector<int>. Gli array C nudi decadono a puntatore: arr equivale a &arr[0]. Non hanno controllo dei bound: accedere fuori range è UB.

Funzioni

CostruttoSintassiSemantica dettagliata
Dichiarazione (prototipo)int foo(int a, int b);Nel file .h. Dice al compilatore che la funzione esiste e quale è la sua firma. Non include il corpo. Il compilatore usa la dichiarazione per verificare le chiamate; il linker poi trova la definizione nel .cpp corrispondente.
Definizioneint foo(int a, int b) { return a+b; }Nel file .cpp. Include il corpo della funzione. Una funzione può avere una sola definizione in tutto il programma (ODR - One Definition Rule), ma può avere quante dichiarazioni si vuole.
Parametri di defaultvoid f(int a, int b=4, int c=3);I parametri con default vanno dichiarati nel .h e NON ripetuti nel .cpp. Devono essere alla fine della lista parametri. Con f(1) i parametri b e c assumono i loro default. Il default viene "inserito" dal compilatore nel punto di chiamata.
Overloadingvoid move(int x, int y);
void move(int d);		Stesso nome, firma diversa (tipo o numero di parametri). Il compilatore sceglie la versione corretta in base agli argomenti (risoluzione dell'overloading). Il tipo di ritorno non basta per distinguere due overload: int foo() e void foo() non possono coesistere. Il Name Mangling permette al linker di distinguerle.
Passaggio per valorevoid f(int x)Viene creata una copia del valore. Modificare x dentro la funzione non altera l'originale. Per tipi piccoli (int, double, bool) è la scelta migliore. Per oggetti grandi (stringhe, vettori) è costoso: evitare.
Passaggio per riferimentovoid f(int& x)La funzione lavora direttamente sull'oggetto originale senza copiarlo. Modifiche a x cambiano il valore originale. Preferito per oggetti grandi che la funzione deve modificare. Se la funzione non deve modificare l'oggetto, usare const reference.
Passaggio per puntatorevoid f(int* x)Simile al passaggio per reference, ma rende esplicito che si lavora con un indirizzo (e permette nullptr). In C++ moderno preferire i riferimenti quando possibile; usare i puntatori quando l'argomento può essere opzionale (nullable).
Funzioni inlineinline int square(int x) { return x*x; }Suggerisce al compilatore di sostituire la chiamata a funzione con il corpo direttamente nel codice chiamante (evitando overhead della chiamata). Il compilatore può ignorare l'hint. Usato soprattutto per funzioni molto brevi definite negli header. I metodi definiti dentro la classe sono implicitamente inline.

Classi: Struttura e Semantica

CostruttoSintassiSemantica dettagliata
Dichiarazione classeclass Nome { ... };Definisce un nuovo tipo. Il punto e virgola finale è obbligatorio (errore comune dimenticarlo). I membri sono private di default. struct è identico a class ma con visibilità public di default.
Lista di inizializzazioneFoo(int a) : membA(a), membB(0) {}I membri vengono costruiti direttamente con i valori forniti, prima del corpo del costruttore. È più efficiente dell'assegnazione nel corpo (che prima costruisce il membro con default e poi lo riassegna). Per i const member e i riferimenti, è l'unica alternativa. L'ordine di inizializzazione segue l'ordine di dichiarazione nella classe, non quello nella lista.
Costruttore explicitexplicit Foo(int x);Impedisce le conversioni implicite. Senza explicit: Foo f = 5; compilerebbe silenziosamente (il compilatore convertirebbe 5 a Foo). Con explicit: solo Foo f(5); o Foo f{5}; sono validi. Regola: qualsiasi costruttore chiamabile con un solo argomento va marcato explicit (norma Google C++ Style).
Costruttore di defaultFoo() {} o Foo() = default;Viene generato automaticamente dal compilatore solo se non si dichiara nessun altro costruttore. Se si dichiara un costruttore parametrico, il costruttore di default sparisce automaticamente. = default lo ripristina esplicitamente (C++11).
Metodo constint getValue() const;Promette di non modificare lo stato dell'oggetto (nessun membro non-mutable viene modificato). Può essere chiamato su oggetti const e su riferimenti const. Il compilatore verifica la promessa: se il metodo tenta di modificare un membro, è un errore. Tutti i getter devono essere const.
Membro staticstatic int count;Esiste una sola copia condivisa da tutte le istanze della classe. Non è legato a un oggetto specifico. Va definito nel .cpp (es. int Foo::count = 0;): la dichiarazione nel .h non alloca memoria. Può essere acceduto con Foo::count senza alcun oggetto.
Membro mutablemutable int cache;Può essere modificato anche da metodi const. Usato per cache, contatori di accesso, mutex: cose che non cambiano lo "stato logico" dell'oggetto ma richiedono modifica fisica. È un'eccezione alla garanzia di immutabilità del const.
thisthis->membroPuntatore implicito all'oggetto corrente, disponible in tutti i metodi non-statici. Ha tipo Foo* const (puntatore costante, non può essere cambiato a puntare ad altro). In un metodo const, ha tipo const Foo* const. Usato per: disambiguare nomi, passare l'oggetto a funzioni esterne, implementare pattern method chaining (return *this;).
Operatore di assegnamentoFoo& operator=(const Foo& rhs)Chiamato in a = b (dove a ed b esistono già). Differ dal copy constructor che viene chiamato in Foo a = b; (a è già in costruzione). L'operatore di assegnamento deve: (1) verificare il self-assignment (if(this == &rhs)); (2) liberare le risorse esistenti; (3) copiare in profondità; (4) ritornare *this per permettere a = b = c.

Ereditarietà e Polimorfismo: Dettaglio Semantico

CostruttoSintassiSemantica dettagliata
Ereditarietà pubblicaclass D : public B {}Stabilisce la relazione Is-A. I membri public di B diventano public in D; quelli protected restano protected. È la forma più comune: l'interfaccia pubblica di B fa parte dell'interfaccia di D. Un puntatore a B può puntare a un oggetto D senza cast (upcast). Un puntatore a D che punta a B richiede dynamic_cast e può fallire (downcast).
Ereditarietà privataclass D : private B {}Relazione Implemented-In-Terms-Of. Tutti i membri di B diventano private in D: l'utente di D non sa né può usare l'interfaccia di B. Non è una relazione Is-A: un puntatore a B non può puntare a D senza cast. Alternativa alla composizione quando si ha bisogno di accedere ai membri protected di B.
Virtual functionvirtual void use();Il compilatore crea una vtable (virtual function table) per la classe: una tabella di puntatori a funzione. Ogni oggetto contiene un puntatore nascosto (vptr) alla vtable della sua classe reale. Quando si chiama ptr->use() su un puntatore base, il compilatore segue il vptr e chiama la versione corretta a runtime. Questo ha un costo: un'indirezione in più rispetto alla chiamata diretta. Una volta virtual, sempre virtual: il metodo rimane virtual in tutta la gerarchia discendente.
Pure virtual / Classe astrattavirtual void f() = 0;Un metodo puro virtuale non ha implementazione nella classe corrente. La classe che lo contiene diventa astratta: non può essere istanziata direttamente (new Base() è un errore). Serve come interfaccia contrattuale: ogni classe derivata concreta deve fornire un implementazione. Se una derivata non implementa tutti i metodi puri, diventa anch'essa astratta.
override (C++11)virtual void use() override;Dice al compilatore: "sto riscrivendo un metodo virtuale della classe base". Il compilatore verifica che esista effettivamente un metodo virtuale con quella firma nella classe base. Senza override, una firma leggermente sbagliata (es. parametro sbagliato) creerebbe silenziosamente un nuovo metodo invece di riuscire nell'override. override va messo solo nella dichiarazione nel .h, non nel .cpp.
final (C++11)class Foo final {} o virtual void f() final;Impedisce l'ereditarietà dalla classe o l'override del metodo. Il compilatore può ottimizzare le chiamate a metodi final de-virtualizzandole (devirtualization). Usato per classi di utilità o metodi critici per le performance.
Distruttore virtualevirtual ~Base();Se si elimina un oggetto derivato attraverso un puntatore base, il distruttore corretto (quello della classe derivata) viene chiamato solo se il distruttore della base è virtual. Senza virtual: viene chiamato solo il distruttore di Base (memory leak o undefined behavior). Regola: se una classe ha una virtual function, il distruttore deve essere virtual.
Costruttori ed ereditarietàDerived(int a) : Base(a) {}Il costruttore della classe derivata deve invocare esplicitamente un costruttore della base nella lista di inizializzazione, se la base non ha costruttore di default. I costruttori non vengono ereditati (pre-C++11); con using Base::Base; (C++11) si possono ereditare. La catena di costruzione va dalla base verso la derivata; quella di distruzione è l'inverso.

Preprocessore, Compilazione e Linker

CostruttoSintassiSemantica dettagliata
Include guard#ifndef NOME_H #define NOME_H ... #endifEvita che un header venga incluso più di una volta nella stessa translation unit, il che causerebbe errori di "definizione multipla". Alternativa moderna (non standard ma universal supportata): #pragma once in cima all'header.
#include <> vs #include ""#include <iostream> vs #include "myfile.h"<>: il preprocessore cerca nei percorsi di sistema (librerie standard e di sistema). "": il preprocessore cerca prima nella directory del file corrente, poi nei percorsi di sistema. Usare "" per i propri header e <> per le librerie esterne.
Translation UnitUn file .cpp dopo il preprocessingL'unità atomica di compilazione. Ogni .cpp viene compilato indipendentemente in un object file (.o). I file .h non vengono compilati direttamente: sono testo che viene incollato dentro i .cpp che li includono. Il linker poi unisce tutti gli object file.
LinkerFase successiva alla compilazioneRisolve i symbolic references: sostituisce ogni chiamata a funzione con l'indirizzo reale della funzione compilata. Errore di linker "undefined reference": la dichiarazione esiste (il compilatore è contento), ma la definizione non è stata trovata. Cause comuni: file .cpp non incluso nel build, nome di funzione scritto diversamente nel .h e nel .cpp.
Name ManglingGestito automaticamenteIl compilatore C++ modifica i nomi dei simboli per includere informazioni sul tipo dei parametri. void move(int x, int y) diventa qualcosa come _Z4moveii. Questo permette l'overloading (due funzioni con lo stesso nome ma parametri diversi hanno nomi mangled diversi). Per interfacciarsi con codice C (che non fa mangling), si usa extern "C" { ... }.
ODR (One Definition Rule)—Ogni entità (funzione, variabile globale, classe) può avere una sola definizione nel programma. Le classi possono avere la stessa definizione in più translation unit (per questo vanno negli header e si usano le include guard). I metodi inline possono avere la stessa definizione in più translation unit.

AspettoStack (Pila)Heap (Mucchio)

DeallocazioneAutomatica: quando la variabile esce dallo scope (fine blocco {})Manuale: con delete. Se dimenticato → memory leak
VelocitàMolto veloce: spostamento semplice del puntatore allo stack (SP)Più lenta: richiede di trovare un blocco libero nell'heap (allocation overhead)
DimensioneLimitata (tipicamente 1-8 MB). Stack overflow se si supera.Limitata solo dalla RAM disponibile
LifetimeLegata allo scope: non si può restituire un puntatore a una variabile locale!Legata alla chiamata a delete: dura finché si vuole
Uso tipicoVariabili locali, parametri di funzione, oggetti di dimensione nota e piccolaOggetti grandi, oggetti con lifetime non determinabile a compile-time, array di dimensione variabile
Esempioint x = 5; (locale) — automaticoint* p = new int(5); — manuale, richiede delete p;
